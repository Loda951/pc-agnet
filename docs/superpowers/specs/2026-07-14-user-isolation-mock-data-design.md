# 用户隔离 Mock 数据脚本设计

## 背景与目标

为 PostgreSQL 本地环境提供一组可重复生成、归属清晰的测试数据，用于验证不同用户之间的订单和聊天会话不会相互泄漏。脚本只服务于本地 Demo 与隔离性测试，不修改数据库结构或生产认证逻辑。

成功标准：

- 创建 5 个可登录且容易辨认的测试用户。
- 每个用户拥有 5 条可追溯订单。
- 每个用户拥有 2 个带示例消息的历史聊天会话。
- 用户通过真实登录接口登录后，只能查询自己的订单和聊天会话。
- 脚本重复执行只更新脚本所属记录，不产生重复数据。
- 任一步骤失败时回滚整个事务，并以非零状态退出。

## 实现方案

新增异步 Python 脚本 `backend/scripts/seed_user_isolation.py`。脚本使用项目现有的 SQLAlchemy 模型、`AsyncSessionLocal` 和 `PasswordHasher`，避免纯 SQL 重复实现密码哈希或绕过应用认证约束。

所有测试用户共享公开的本地 Demo 密码 `isolation-demo-password`。脚本会在执行摘要中输出该明文密码，但写入 `user_auth_credential.password_hash` 的仍是现有登录逻辑可识别的 PBKDF2 哈希。直接存储明文会导致 `PasswordHasher.verify_password()` 无法解析，测试账号也无法登录。

脚本不直接插入 `user_session`。认证会话必须由 `/api/auth/login` 在实际登录时创建，才能真实覆盖 refresh token 哈希、有效期和状态处理。

## 数据与依赖顺序

脚本在一个数据库事务中按以下顺序执行：

1. 用户：按 `login_identifier` 创建或更新 `test_user_001@example.com` 至 `test_user_005@example.com`，显示名分别为 `test_user_001` 至 `test_user_005`。
2. 登录凭证：按用户 ID 创建或更新认证凭证，使全部用户能够通过真实登录接口登录。
3. 商品依赖：优先复用第一个启用的 SKU；若数据库没有商品，则创建一组专用于隔离测试的最小商品目录数据。
4. 订单：取得已经生成的用户 ID 后，为每人创建或更新 5 条订单以及对应订单项。
5. 聊天会话：为每人创建或更新 2 个 `conversation`，每个会话包含一条用户消息和一条助手消息。
6. 摘要：事务提交成功后，输出用户、订单和聊天会话的完整归属摘要。

用户始终先于订单和会话写入，订单和会话的 `user_id` 只取自本事务中已经查询或创建成功的 `AppUser.id`。

## 可追溯性与幂等性

仓库现有 `order_info` 表没有 `order_no` 或 `remark` 字段，因此不新增 migration，而是使用预留的确定性 BIGINT 主键作为可读订单号：

- 订单 ID 格式：`991000000000 + 用户编号 * 100 + 订单序号`。
- 会话 ID 格式：`992000000000 + 用户编号 * 100 + 会话序号`。
- 消息和订单项同样使用独立的确定性 ID 段。
- `receiver_name`、`receiver_address`、会话标题和消息内容中冗余 `test_user_NNN`，便于人工核查归属。

测试用户通过唯一的 `login_identifier` 幂等定位。订单、订单项、会话和消息通过确定性主键幂等定位。已有脚本记录会被更新为声明的数据；不会先清空业务表，也不会影响非脚本数据。

若预留主键已被非对应用户的记录占用，脚本立即报错并回滚，而不是覆盖可能属于其他测试或业务的数据。

## 业务字段

订单覆盖以下字段：

- `total_amount`、`pay_amount`、`freight_amount`
- `pay_type`
- 待付款、待发货、已发货、已完成、已关闭等状态
- 收件人姓名、测试手机号和地址
- 创建、支付和发货时间；尚未发生的时间保持为空
- 商品名称、规格、单价和数量

金额、状态和时间使用确定性规则生成，使不同用户和不同订单既有区别又便于重复核查。

## 事务与错误处理

- 使用单个 `session.begin()` 包裹全部写入。
- 数据库异常或归属冲突会触发自动回滚。
- 顶层捕获异常并向标准错误输出失败原因，进程返回非零退出码。
- 只有事务提交成功后才打印“完成”摘要，避免把已回滚数据误报为成功。

## 输出格式

摘要分为三段：

- 用户：`user_id`、显示名、登录标识和公共 Demo 密码。
- 订单：订单 ID、`user_id`、用户名、金额、状态和创建时间。
- 聊天会话：会话 ID、`user_id`、用户名和标题。

输出按用户编号和记录序号稳定排序，方便复制到人工测试记录中。

## 验证策略

单元测试不依赖 PostgreSQL，直接测试脚本中负责构造确定性数据和检查记录归属的纯函数，覆盖：

- 5 个用户及其标识生成正确。
- 每个用户生成 5 个不重复、包含用户编号的订单 ID。
- 状态对应的支付和发货时间合理。
- 同一输入重复生成相同的确定性标识。
- 归属检查函数在固定 ID 属于其他用户时抛出异常。

静态验证执行：

```bash
cd backend
pytest tests/test_seed_user_isolation.py
ruff check scripts/seed_user_isolation.py tests/test_seed_user_isolation.py
```

若本地 PostgreSQL 已启动，再执行：

```bash
cd backend
alembic upgrade head
python -m scripts.seed_user_isolation
```

人工隔离验证流程为：先分别使用两个测试账号调用登录接口，再查询订单列表、指定订单和聊天会话列表；账号 A 的确定性订单或会话 ID 对账号 B 应返回 `404` 或不出现在列表中。

## 非目标

- 不修改认证算法或允许数据库明文密码。
- 不修改 Alembic migration 或给订单表新增字段。
- 不自动调用 API 执行完整端到端隔离测试。
- 不清理其他 Demo、测试或真实业务数据。
