# 导入真实商品数据并强化推荐与集成测试

## 背景与目标

- 当前 demo seed 商品量较小，商品推荐容易因为数据少或规格不完整而返回空结果。
- `docyx/pc-part-dataset` 已提供 `mouse.json`、`keyboard.json`、`headphones.json` 等真实 PCPartPicker 外设数据。
- 本 feature 的目标：
  - 支持从真实数据集目录批量导入 mouse、keyboard、headphones。
  - 规范外设关键属性，让筛选、排序、对比和兼容性解释有足够上下文。
  - 补齐 FastAPI + Repository + Agent 的可回归集成测试，避免后续改 schema、prompt 或检索逻辑时回归。

## 关键变更清单

### 数据集 Mapper

- `backend/app/services/dataset_mapper.py`
  - 扩展 `CATEGORY_NAME_MAP`，覆盖 mouse/mice、keyboard/keyboards、headphones/headset/headsets 等别名。
  - 扩展品牌识别，新增 SteelSeries、Beyerdynamic、Sennheiser、Audio-Technica、Sony、HiFiMAN 等真实样本中常见品牌。
  - 扩展 `SPEC_ATTRIBUTES` 和 `FILTER_ATTRIBUTES`：
    - 鼠标：`tracking_method`、`connection_type`、`max_dpi`、`hand_orientation`、`color`
    - 键盘：`style`、`switches`、`backlit`、`tenkeyless`、`connection_type`、`color`
    - 耳机：`type`、`frequency_response`、`microphone`、`wireless`、`enclosure_type`、`color`
  - 保留布尔字段为中文展示值 `是/否`，让耳机 `wireless`、`microphone` 可直接用于筛选和回答。
  - SKU 标题只追加颜色和轴体等稳定变体字段，避免把长连接方式写进标题导致噪声。

### 真实数据导入脚本

- `backend/scripts/import_pc_part_dataset.py`
  - 支持传入单个 JSON/JSONL 文件，也支持传入 `data/json` 目录。
  - 单文件模式可从文件名推断 part type，例如 `mouse.json` -> `mouse`。
  - 目录模式默认导入 `headphones`、`keyboard`、`mouse`，也可通过 `--part-types` 指定。
  - `--limit` 改为可选的每文件上限，不传时导入匹配文件中的全部有效记录。

示例：

```bash
cd backend
python -m scripts.import_pc_part_dataset ../pc-part-dataset/data/json
python -m scripts.import_pc_part_dataset ../pc-part-dataset/data/json --part-types mouse,keyboard,headphones --limit 500
```

### 商品搜索与推荐解释

- `backend/app/repositories/catalog.py`
  - 搜索候选从 SKU/SPU/品牌/分类扩展为同时读取 EAV 属性关系。
  - 商品返回的 `specs` 会合并 `sku.specs_json` 和属性关系，避免真实导入后筛选字段缺失。
  - 查询从整句 `LIKE` 改为 token 化匹配和排序，支持“G502 Viper 对比”这类多商品对比查询。
  - `connection_type=Wireless` 可匹配：
    - 鼠标/键盘的 `connection_type` 包含 Wireless
    - 耳机的 `wireless=是`
  - 排序综合标题、品牌、分类、规格命中、筛选命中、库存和价格。

- `backend/app/agent/graph.py`
  - fallback 商品推荐回答新增简短“适配提示”：
    - 鼠标说明连接方式、DPI 和握持方向。
    - 键盘说明连接方式、轴体和布局。
    - 耳机说明无线、麦克风和封闭/开放类型。

- `backend/app/agent/intent.py`
  - 商品搜索 query 清理增加对比、比较、哪款等噪声词。
  - 支持用户说“麦克风/带麦”时生成 `microphone=是` 筛选。

### Upsert 行为

- `backend/scripts/seed_demo.py`
  - 已有 SPU/SKU 再次导入时会更新品牌、价格、库存、规格和状态。
  - 已有属性键会按新规则补强 `is_spec` / `is_filter` 标记。

### 集成测试

- `backend/tests/conftest.py`
  - 新增 PostgreSQL 集成测试 fixture：
    - 每个测试创建独立 `NullPool` async engine，避免 asyncpg 跨事件循环复用。
    - 外层事务包裹 seed 和 API 请求，测试结束统一 rollback。
    - 数据库不可用时跳过集成测试，避免无基础设施环境误失败。

- `backend/tests/test_catalog_repository.py`
  - 覆盖无线耳机通过 `wireless=是` 被 `connection_type=Wireless` 筛中。
  - 覆盖对比查询按 G502/Viper 等独立 token 命中多个商品。

- `backend/tests/test_api_integration.py`
  - 覆盖 `/api/chat` 商品推荐，不依赖真实 LLM key。
  - 覆盖 `/api/chat` 售后政策 RAG evidence，使用 fake knowledge service。
  - 覆盖 `/api/orders/latest`。
  - 覆盖 `/api/after-sales` 返回 `409 + human_handoff_required`。

- `backend/tests/test_import_pc_part_dataset.py`
  - 覆盖文件名推断、JSON/JSONL 读取、目录默认三类外设解析。

## 方案与决策

- 决策：暂不新增属性标准化表。
  - 理由：现有 `attribute_key` / `attribute_value` / `goods_attribute_relation` 已能承载真实数据筛选；本阶段通过 mapper 和 repository 规范化即可。

- 决策：目录导入默认只处理 headphones、keyboard、mouse。
  - 理由：阶段需求聚焦核心外设，避免一次性导入完整 66k+ PC 零件数据影响本地 demo。

- 决策：商品搜索排序先在 repository 内实现轻量 token 权重。
  - 理由：当前数据规模和 MVP 需求还不需要引入外部搜索引擎；保留 SQLAlchemy 和现有 schema 即可回归。

- 决策：集成测试使用真实 PostgreSQL，但 LLM 和知识检索外部依赖保持 mock/fake。
  - 理由：订单、商品、会话持久化需要真实数据库路径；LLM 和 Chroma 不应成为回归测试的不稳定因素。

## 遗留事项

- 商品、订单、物流事实仍未统一纳入 `evidence` schema。
- 搜索排序目前是轻量规则权重，后续可结合点击/转化或离线评测集继续调整。
- 集成测试在没有 PostgreSQL 基础设施的环境中会跳过，需要 CI 时配套启动测试数据库。
