# 多用户鉴权、会话管理与用户隔离

## 背景与目标

- 第二阶段的核心风险是订单、会话、记忆和售后上下文仍可能由 demo 默认用户或公开 `user_id` 决定。
- 本 feature 的目标：
  - 引入真实登录态、refresh session 和退出登录。
  - 所有敏感业务入口只信任当前认证用户。
  - 前端支持登录、会话恢复、登录过期处理和退出登录。
  - 测试覆盖用户间订单、会话、记忆和人工接管记录隔离。

## 关键变更清单

### 数据库与模型

- 新增 migration `0002_user_auth_sessions`：
  - `app_user` 增加 `login_identifier`、`status`、`last_login_at`、`updated_at`。
  - 新增 `user_auth_credential` 保存登录标识和密码哈希。
  - 新增 `user_session` 保存 refresh token 哈希、状态、过期时间和撤销时间。
- 新增 ORM 模型：
  - `UserAuthCredential`
  - `UserSession`

### 后端认证

- 新增 `AuthService`：
  - PBKDF2-SHA256 密码哈希与校验。
  - HMAC access token 签发与校验。
  - refresh token 只保存 SHA-256 哈希，并在刷新时轮换。
  - 登录、刷新、登出和当前用户解析。
- 新增 `/api/auth/login`、`/api/auth/refresh`、`/api/auth/logout`、`/api/auth/me`。
- 新增 `get_current_user` FastAPI 依赖，统一处理 `401`、`403`。
- `scripts.seed_demo` 写入本地演示账号：
  - 登录标识：`demo@example.com`
  - 密码：`demo-password`
- 默认 seed 同时写入 5 个多用户 Mock 账号：
  - 登录标识：`test_user_001@example.com` 至 `test_user_005@example.com`
  - 统一密码：`isolation-demo-password`
  - 每个账号 5 笔独立订单，状态覆盖待付款、待发货、已发货、已完成和已关闭
  - 订单从 active SKU 池中按固定种子稳定抽样，商品、金额、时间、地址和物流均可区分

### 用户隔离

- `/api/chat` 和 `/api/chat/stream` 不再读取请求体 `user_id`。
- `ChatRequest` 删除 `user_id`，并对额外字段使用 `extra="forbid"`。
- `/api/orders/latest` 和 `/api/orders/{order_id}` 删除公开 `user_id` query 参数。
- `/api/after-sales` 入口全部要求认证用户。
- 新增只读售后记录查询接口，按当前用户过滤：
  - `GET /api/after-sales`
  - `GET /api/after-sales/{ticket_id}`

### 前端

- `frontend/src/api.ts` 集中管理 access token、refresh token、刷新重试和本地清理。
- 新增登录页和登录恢复流程。
- 工作台左侧展示当前登录用户，并提供退出登录。
- 聊天请求遇到 `401/403` 时清理本地会话并回到登录页。

## 方案与决策

- 决策：不新增第三方 JWT/passlib 依赖，先使用标准库 PBKDF2 和 HMAC token。
  - 理由：当前阶段需要本地 demo 和测试稳定，标准库足够满足可撤销 session 闭环。
- 决策：refresh token 只存哈希，并在刷新时轮换。
  - 理由：避免数据库泄露后 refresh token 可直接使用。
- 决策：`ChatRequest.user_id` 彻底删除并拒绝额外字段。
  - 理由：身份来源必须唯一来自认证依赖，不保留公开绕行路径。
- 决策：售后办理仍保持 read-only 降级，不自动创建真实办理工单。
  - 理由：人工接管队列是下一阶段功能，本阶段只补认证和已有记录的用户隔离。

## 遗留事项

- 当前 access token 是项目内 HMAC 格式，不是标准 JWT；后续如接入外部网关或第三方身份源，可替换 token provider。
- 人工接管仍未升级为真实队列，只补了认证后的只读隔离入口。
- 集成测试需要 PostgreSQL 基础设施运行时才会执行数据库隔离路径。
