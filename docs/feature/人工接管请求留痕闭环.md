# 人工接管请求留痕闭环

## 背景与目标

- 此前 `human_handoff_required` 只在聊天和前端右侧面板提示，不会产生后端可追踪记录。
- 当前阶段目标是形成“创建 -> 存储 -> 查询”的最小闭环，但不接入真实人工客服系统。
- 本功能只记录用户诉求和当前状态，不自动办理退款、退货、维修或订单修改等业务操作。

## 关键变更

### 数据模型

- 新增 migration `0003_handoff_requests`。
- 新增表 `handoff_request`：
  - `id`
  - `user_id`
  - `session_id`
  - `order_id`
  - `request_type`
  - `reason`
  - `boundary_category`
  - `status`
  - `created_at`
  - `updated_at`
- `request_type` 限定为 `refund`、`return`、`repair`、`order_change`、`other`。
- `status` 限定为 `pending`、`acknowledged`、`resolved`，当前创建时固定为 `pending`。

### 后端接口

- `POST /api/after-sales` 改为创建 `handoff_request`，成功返回 `202 Accepted`。
- 响应体包含：
  - `request_id`
  - `status`
  - `message`
- `message` 明确说明请求已记录，但系统不会自动办理业务操作。
- 新增 `GET /api/after-sales/handoff-requests/{request_id}`，按当前认证用户查询记录。
- 创建时校验 `session_id` 属于当前用户；`order_id` 若提供，也必须属于当前用户。
- 保留旧 `after_sales_ticket` 查询接口，不把人工接管留痕写入旧售后工单表。

### 前端交互

- 右侧接管面板展示待记录和已记录状态。
- 用户点击“记录人工确认诉求”后调用 `POST /api/after-sales`。
- 创建成功后展示请求编号、当前状态和固定说明文案：

```text
当前系统不会自动办理退款、退货、维修或订单修改等业务操作，请求已记录待人工确认
```

- 接管面板提供请求编号输入框，可调用查询接口查看状态。

## 边界约束

- 不接入真实人工客服系统。
- 不派发工单，不调用退款、退货、维修、订单修改等下游业务。
- 当前不提供状态流转接口，只预留 `pending -> acknowledged -> resolved` 状态空间。
- 建议动作只打开待记录状态，不自动创建接管记录。

## 验证结果

- 后端边界测试覆盖 `POST /api/after-sales` 的 `202 + request_id + pending` 契约。
- 后端 API 集成测试覆盖创建后按 `request_id` 查询。
- 用户隔离测试覆盖用户 B 不能查询用户 A 的接管记录。
- 前端构建覆盖新增 API 类型、创建按钮和查询入口。
